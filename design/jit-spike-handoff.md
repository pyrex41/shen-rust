## 0. The scoreboard (why we're here)

`scripts/cross-port-bench.sh` times the 134-test kernel suite against the
reference, **shen-cl (SBCL, native-compiled)**:

| | kernel-tests (134/0) | ratio |
|---|---|---|
| **shen-cl (SBCL)** | ~2.0 s (loaded machine) / **~1.0 s cool** | **1.0√ó (target)** |
| **shen-cedar (post-flip, post-dispatch-fix)** | ~7.2 s loaded / ~5.0 s cool | **~3.55√ó slower** |

> Absolute numbers drift ¬±5‚Äì12% with thermal/load state ‚Äî **trust the paired
> ratio, not the absolute** (this machine has been running hot; the same
> baseline binary measured 5.0s cool and 10.7s loaded). Always run shen-cedar
> and shen-cl back-to-back, min-of-N, on the same machine state.

**The profile says the gap is interpreter dispatch, not real work.** A `sample`
of the kernel-tests worker thread is dominated by **`aot::runtime::call_or_apply`
(AOT-to-AOT / CPS-continuation dispatch)** and **`interp::eval::eval_in` +
`Interp::apply` (the tree-walked continuations)**, wrapped around the actual
type-check work (`prolog::lzy=!`, `t_star::system`, `prolog::bind!`). SBCL is
~3.5√ó faster because it native-compiles that dispatch away. **You cannot
profile-tune a 3.5√ó out of an interpreter loop** ‚Äî micro-opts net ~5% (the last
commit, `c164fdd`, cut the closure-dispatch thread-local tax: 3.72√ó‚Üí3.55√ó). The
structural lever is to **stop interpreting the hot path and run native code**.

## 1. Why a JIT, and why *now*

The ladder (`perf-state-and-gc-ladder.md`): **tracing-GC infra ‚Üí word-sized
`Copy` Value ‚Üí Cranelift JIT**. The first two rungs are **shipped** (GC Step 1/2
landed `src/gc/`; Step 3 flipped `Value` to a `#[repr(transparent)] u64` `Copy`
word over the GC heap ‚Äî commits `f0beb2f`/`12c3cc1`/`1cbc69f`/`c164fdd`).

The prior Cranelift research (in memory) was decisive about ordering: a JIT on
the **old 24-byte `Rc`-laden `Value`** was worth only ~1.5√ó because JIT'd code
**cannot hold an aggregate `Value` in registers** ‚Äî every clone/drop/arith/cons
would FFI back into Rust to refcount. The 3‚Äì8√ó that approaches SBCL **requires a
word-sized `Value` first**, so generated code can inline arithmetic + tag
dispatch and keep values in registers, FFI-ing to Rust only for genuinely
heap-touching ops (cons alloc, apply, string ops).

**That prerequisite is now paid.** `Value` is an 8-byte `Copy` word; the low-3-bit
tag scheme (`value.rs`) lets JIT'd code test "is fixnum?" / "is this tag?" and do
fixnum arithmetic with plain integer ops + a shift, no FFI. This is the moment
the JIT pays off.

## 2. The architecture you're slotting into

**Today, two execution engines feed one heap:**
- **AOT (the default, dominant on kernel-tests):** `klcompile`
  (`crates/klcompile/`) lowers each kernel `.kl` file to **Rust source at build
  time** (`crates/shen-cedar/src/aot/kernel/*.rs`, regenerated by
  `scripts/codegen-kernel-aot.sh`). Each AOT function is a Rust `fn(&mut Interp,
  &[Value]) -> ShenResult<Value>`. AOT-to-AOT calls go through
  `rt::apply_direct(interp, name, args)` ‚Üí a **raw fn-pointer "direct table"**
  (`interp.get_aot_direct(sym)`) on hit, else `apply_named` ‚Üí `call_or_apply`.
- **Tree-walker / bytecode VM:** runtime `defun`/`lambda`/`freeze` (and `eval-kl`)
  run interpreted. The VM (`vm/exec.rs`) is **opt-in** (`SHEN_CEDAR_VM=1`) and
  does NOT move kernel-tests (see ladder ¬ß3) ‚Äî *do not* base the JIT on it.

**Key dispatch seam** (`aot/runtime.rs`): `call_or_apply` resolves a `Value`
closure and calls `ClosureKind::Native(nf, _)` / `Bytecode(..)` / `Lambda(..)`.
The CPS continuations the type-checker builds with `make_aot_closure` are
`ClosureKind::Native` closures ‚Äî **these are the hot callees** and they can't use
the `apply_direct` fast path (they're runtime closure *values*, not statically
named). That is exactly the dispatch the JIT should subsume.

**The JIT tier-up shape (recommended):** add a `ClosureKind::Jit(JitFn, ‚Ä¶)` (a
raw finalized-code pointer from Cranelift) and/or populate the `aot_direct` table
with JIT'd fn pointers. `call_or_apply` then calls JIT'd native code directly,
same calling convention as `Native`. JIT'd code:
- holds `Value`s as `i64`/`i64`-tagged words in registers;
- inlines fixnum arith + tag tests + `nil`/`bool`/`cons` checks;
- **calls into the Rust runtime** via `extern "C"` for: cons/vec/str/closure
  allocation (`Value::cons` etc., which use the thread-local heap), `apply` of a
  non-JIT callee, primitives it doesn't inline, error construction.

## 3. Do a SPIKE first (project discipline ‚Äî every prior rung was spiked)

Every rung of this ladder was de-risked with a standalone `benches/*_spike.rs`
before any productionization (`value_repr_spike`, `gc_spike`,
`gc_roots_aot_spike`). **Do the same.** Build `benches/jit_spike.rs`
(`harness = false`, paired-alternating min-of-N, no external bench deps beyond
the `cranelift-*` crates) that:

1. Takes **one representative hot shape** ‚Äî start with a self-recursive
   arithmetic function (e.g. a `fib`/`sumto`/`count-down` analogue over fixnums)
   so most ops inline and the FFI surface is just the recursive call + maybe
   cons. This isolates "does native codegen beat the interpreter on a realistic
   shape" with minimal runtime plumbing.
2. JIT-compiles it with Cranelift to a real `fn(*mut Interp, *const Value, usize)
   -> Value` (or your chosen ABI), calling back into Rust runtime helpers for the
   ops it doesn't inline.
3. Times it against the **same function run through the existing AOT/tree-walker
   path** (install it as a normal Shen `defun` and call via `interp.apply`),
   paired min-of-N, on identical inputs, asserting identical results.

**Kill-criterion (state it up front, like the GC spikes):** the JIT path must
**materially beat** the interpreted path on the representative shape *after*
accounting for the Rust-FFI calls it still makes (cons/apply). If a fixnum-heavy
recursive function JITs to ~native and beats the interpreter by a wide margin,
the lever is real ‚Üí productionize. If the FFI-per-cons tax dominates and the win
is marginal, that is a critical finding (re-scope: inline cons too, or tier only
arith-heavy leaves). **A spike that "passes" only because the hard case wasn't
exercised is a void result** ‚Äî make the workload force real recursion + at least
some heap allocation, and assert correctness (see the gc-spike measurement
lessons in memory: aligned caps / no-assert runs gave false 7√ós).

## 4. Cranelift specifics (from the research; verify against current crates)

- Crates: `cranelift-codegen`, `cranelift-frontend`, `cranelift-jit`,
  `cranelift-module` (was ~0.132, May 2026 ‚Äî check latest). `cranelift-jit`'s
  `JITModule` + `JITBuilder`, define a `Function`, `module.finalize_definitions()`,
  `get_finalized_function(id)` ‚Üí a raw code pointer you `transmute` to your fn ABI.
- **No aggregate types** ‚Äî Cranelift works on machine words only. **Fine now**:
  `Value` is one `i64`. Pass/return it as `types::I64`. `Interp`/args pass as
  pointers (`types::I64` pointer-width).
- **Tail calls** for Shen's pervasive TCO: `CallConv::Tail` + `return_call` /
  `return_call_indirect` are supported and default-capable on **aarch64 (Apple
  Silicon)** ‚Äî caller+callee must both use the Tail conv; `extern "C"` runtime
  calls stay non-tail. This is how you avoid blowing the stack on deep
  tail-recursion without the 1 GB worker-stack hack (`bin/shen-cedar/src/main.rs`).
- Runtime calls: declare each Rust helper (`rt_cons`, `rt_apply`, `rt_alloc_*`,
  `rt_error`, ‚Ä¶) as an imported symbol in the `JITBuilder` (`symbol("rt_cons",
  rt_cons as *const u8)`); call via `func.declare_func_in_func` + `call`.
- Keep the helpers `extern "C"` with a stable ABI; pass `Value` as `u64` and
  `*mut Interp` through.

## 5. The hard problems ‚Äî address them in the spike, not after

1. **GC roots in JIT'd frames (the load-bearing one).** JIT'd code holds live
   `Value`s in registers and Cranelift-managed stack slots. The GC's Step-4 root
   story (`perf-state-and-gc-ladder.md` ¬ß6, and the **gc_roots_aot_spike** which
   PASSED: conservative native-stack scan `[sp..stack_base)` + aarch64
   callee-saved register flush, sound because the heap is non-moving ‚áí a
   false-positive root only over-retains). **JIT frames are ordinary native
   frames** ‚Äî the conservative scan *should* reach them ‚Äî but **verify**: a
   register-resident `Value` not spilled at a collection safepoint is invisible
   to a stack scan. In the spike, collection is OFF (grow-only, Step 3), so this
   can't crash the spike ‚Äî but you MUST reason about it before productionizing,
   because Step 4 turns collection on. Likely answer: ensure `Value`s are spilled
   to the (scannable) stack across any call that can trigger collection, or
   register JIT code regions with the conservative scanner. This is the #1 thing
   to get right and the reason a moving GC was rejected (¬ß6).
2. **Where the JIT gets the heap.** `Value::cons` etc. use the **thread-local**
   `HEAP` (`value.rs`). JIT'd code calls a runtime `rt_cons` that does the same ‚Äî
   no new heap plumbing needed. (If `HEAP.with` shows up hot, the same
   "pass the heap via `Interp`" revisit applies to both Rust and JIT paths.)
3. **Tier-up trigger + code cache.** Decide what to JIT and when: simplest is
   compile-on-install for a chosen set of hot kernel functions (offline-ish, like
   AOT), or a call-count threshold for runtime tier-up. Cache finalized code;
   never re-JIT the same body (cf. the VM's ~1.2M-recompiles bug, memory).
4. **Correctness oracle.** Reuse the discipline of `tests/vm_differential.rs`:
   any JIT'd function must produce `shen_eq`-equal results (and matching Ok/Err)
   to the interpreted version, across a corpus. Wire a JIT differential test
   before trusting it.

## 6. Explicit boundaries (what this spike is NOT)

- **Not** turning GC collection on ‚Äî that's **GC Step 4** (register precise roots
  + conservative AOT/JIT-frame scan + lower the cap). Orthogonal; the JIT spike
  runs fine grow-only. But note the interaction in ¬ß5.1: Step 4's root set must
  eventually include JIT frames.
- **Not** based on the opt-in bytecode VM (it doesn't move kernel-tests).
- **Not** LLVM or Wasm ‚Äî both rejected with reasons (memory
  `arch-execution-engine`: LLVM compile-latency wrong for JIT; Wasm host-boundary
  marshaling fatal for this callback-heavy, shared-heap workload).
- **Not** "JIT everything" ‚Äî start with the hottest, simplest shape and grow.

  actually exercised (real recursion + real allocation), not just "result

## 9. Suggested first move

Start the spike with a **fixnum-only self-recursive function** (no allocation):
JIT `count-down`/`sumto` with `CallConv::Tail` + `return_call`, fixnum arith
inlined, zero FFI. If *that* doesn't beat the interpreter by a wide margin,
something is wrong with the harness ‚Äî fix it before adding heap ops. Then add
**one** allocating shape (cons-building) to measure the FFI-per-alloc tax
honestly. Those two data points tell you whether the JIT closes the 3.55√ó or
whether cons-FFI is the next wall (‚Üí inline allocation, or tier only arith
leaves). Report both numbers with the kill-criterion verdict before
productionizing.

## 10. SPIKE RESULT (2026-05-29) ‚Äî kill-criterion PASSED, lever is real

`benches/jit_spike.rs` (Cranelift 0.132, harness=false, paired min-of-N,
PAIRS=10, all engines asserted `shen_eq`-equal). Three shapes √ó three engines:
**tree** (tree-walked `Lambda` via `Interp::apply`), **aot** (a `DirectFn`
recursing through the real `intern_static`‚Üí`get_aot_direct`‚Üíindirect-call seam ‚Äî
the dispatch the profile is dominated by), **jit** (Cranelift native on the
`Value` word). Run on a 1 GiB-stack thread (the `aot` baseline is genuinely
Rust-recursive, like real AOT leaning on the binary's big stack).

| shape | tree ms/call | aot ms/call | jit ms/call | **JIT vs aot** | JIT vs tree |
|---|---|---|---|---|---|
| `fib(30)` ‚Äî non-tail, fixnum, shallow stack | 254.5 | 9.59 | 2.77 | **3.46√ó** | 91.8√ó |
| `sumto-acc(200000)` ‚Äî self-tail (return_call TCO) | 17.7 | 1.79 | 0.050 | **35.67√ó** | 353√ó |
| `build(2000)` ‚Äî one cons/step (rt_cons FFI) | 0.195 | 0.029 | 0.013 | **2.28√ó** | 15.1√ó |

**Verdict: PASS (‚â•2√ó over aot-dispatch on fib ‚Üí productionize).** The JIT is the
structural lever the scoreboard demanded: shen-cedar is ~3.55√ó off SBCL and the
gap is *dispatch* (`call_or_apply`/`eval_in`), not real work ‚Äî and JIT'd native
code removes 3.46√ó of that dispatch cost on the hot fixnum path.

**De-risked by the spike (these were the open questions):**
- **`CallConv::Tail` + `return_call` self-recursion works on aarch64/Cranelift
  0.132** ‚Äî sumto recursed 200k deep at *constant* stack (0.050 ms/call). This is
  the mechanism Shen's pervasive (and mutual) tail recursion needs.
- **A default-callconv (host C ABI) entry trampoline can `call` a tail-callconv
  callee** ‚Äî the Cranelift verifier accepts it; this is how Rust calls into the
  tail-recursive JIT code. (Pattern: `define_entry` ‚Üí normal `call` ‚Üí tail fn.)
- **The `Value` word + `rt_cons` FFI path is correct** ‚Äî all three engines built
  `shen_eq`-identical lists; immediates and heap cons round-trip through the JIT.
- **Allocation is not a wall.** The cons-FFI tax narrows the win (3.46‚Üí2.28 vs
  aot) but the JIT still beats dispatch with a heap op per step. Inlining
  allocation (bump-in-JIT) would widen it but is *not* required to win.

**Honest caveats / what the spike did NOT prove (address in productionization):**
- **Fixnum arith was raw-word** (no tag-check / overflow-to-float guard). Real
  codegen adds a fixnum tag test + `checked_add` overflow path (a couple of
  predicted branches) ‚Äî will shave some of the 3.46√ó, but there is large margin.
- **GC roots in JIT frames (¬ß5.1) is still UNVERIFIED.** The spike ran the heap
  **grow-only** (Step 3; collection off), so a register-resident `Value` not
  spilled at a safepoint could be missed once Step 4 turns collection on. JIT
  frames are ordinary native frames so the conservative scan *should* reach them,
  but verify register-spill-at-safepoint before relying on it. **This is the #1
  gate before Step 4 + JIT coexist.**
- **Microbenchmark on isolated shapes.** Real kernel-tests speedup depends on the
  JIT-able fraction and on the cross-function (non-self) tail-call / apply-of-
  closure paths, which still FFI back to the runtime. The 3.46√ó is the *ceiling*
  for the pure-dispatch component, not a whole-suite multiplier.

**Next (productionization sketch):** add `ClosureKind::Jit(JitFn, ‚Ä¶)` and/or
populate `aot_direct` with JIT'd fn pointers; tier the hottest kernel/continuation
shapes; emit fixnum tag+overflow guards; wire a JIT differential oracle
(`tests/vm_differential.rs` pattern); then settle the ¬ß5.1 safepoint-spill story
before Step 4. `benches/jit_spike.rs` is the regression anchor.
